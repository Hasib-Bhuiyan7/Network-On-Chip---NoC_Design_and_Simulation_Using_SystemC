# Network-On-Chip---NoC_Design_and_Simulation_Using_SystemC
# //SystemC-Based 4×4 Mesh Network-on-Chip (NoC) Design//

This project focuses on the design and simulation of a Network-on-Chip (NoC) using SystemC, starting from a simple 1×2 configuration and extending it into a 4×4 mesh topology. The main objective was to understand how on-chip communication works at a system level, and then apply that understanding to build a scalable network by reusing and connecting existing modules. Rather than redesigning components, the work mainly involved analyzing the provided architecture and carefully expanding it while maintaining correct functionality.

The overall system is built using three main modules: source, router, and sink. Each router is designed with five ports corresponding to the four cardinal directions (North, South, East, West) along with a local connection to its associated IP core. Data is transmitted in the form of flits, where a sequence of flits implicitly represents a packet. In this implementation, packets are not explicitly grouped as a single structure; instead, the transmission is handled at the flit level, with the final flit marked using a tail bit. This follows the concept of wormhole routing, where the path is determined at the beginning of transmission and maintained until the entire packet has passed through.

A large part of the design revolves around the router, specifically the arbiter and crossbar modules. The arbiter is responsible for deciding which input gets access to which output when multiple requests occur at the same time. It checks each input buffer, determines whether data is available, and then performs routing based on the destination address. The routing logic follows a simple deterministic approach (XY routing), where the current router’s position is compared with the destination to decide whether the flit should move horizontally or vertically. Once a path is selected, the arbiter grants access and reserves the output port. This reservation remains active until the tail flit is detected, at which point the connection is released. This behavior is important because it ensures that all flits belonging to the same packet follow the same path without interruption.

The crossbar works alongside the arbiter by physically forwarding the data from the selected input port to the corresponding output port. While the arbiter handles the decision-making, the crossbar handles the actual movement of flits. Each input port is also connected to a FIFO buffer, which temporarily stores incoming flits. This buffering helps manage differences in timing and prevents data loss when multiple flits arrive simultaneously.

The transition from a 1×2 NoC to a 4×4 mesh is mainly handled in the main SystemC file. Instead of manually connecting each router, a structured approach is used where routers are instantiated in a loop and connected based on their position in the grid. Each router is assigned an ID, and its row and column are calculated using simple indexing. From there, connections are made conditionally: routers connect to their east neighbor if they are not on the right edge, to their west neighbor if they are not on the left edge, and similarly for north and south connections. This approach avoids hardcoding connections and makes the design much easier to scale.

In terms of functionality, different traffic patterns were tested to verify that the network behaves correctly. In the uniform pattern, each source sends data to a unique destination, ensuring that no two sources target the same sink. In the neighbouring pattern, each source communicates only with an adjacent node. These patterns help demonstrate both long-distance routing across the mesh and short, local communication. The simulation output, which is printed using standard output statements, shows packets being generated at the source and successfully received at the correct sink. The messages include details such as source ID, destination, and data values, making it easy to track how packets move through the network.

From the results, it can be observed that the routing logic works as expected, with packets reaching their intended destinations even when multiple routers are involved. The arbiter successfully handles contention between inputs, and the wormhole routing mechanism ensures that once a path is established, flits continue flowing without unnecessary delays. There are minor effects at the start of simulation, such as initial flits not appearing immediately at the sink, which can be attributed to synchronization and initialization timing. However, once the system stabilizes, communication proceeds consistently.

Overall, this project demonstrates how a relatively small NoC design can be expanded into a larger and more complex system by focusing on modularity and proper interconnection. It reinforces key concepts such as routing, arbitration, buffering, and flow control, while also showing how SystemC can be used to model hardware-level behavior at a higher level of abstraction. The final 4×4 mesh design serves as a working example of a scalable NoC and provides a solid base for further experimentation or optimization.
